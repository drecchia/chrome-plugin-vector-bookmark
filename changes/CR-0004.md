# CR-0004 — Banner de variáveis de ambiente no startup do daemon

**Data:** 2026-05-02
**Solicitante:** Danilo Recchia
**App Version:** daemon 0.1.0
**Status:** implemented
**Urgência:** low
**Domínios afetados:** Daemon startup (DX/observabilidade)

---

## Descrição da mudança

No startup do daemon (`vbmd server`), imprimir um **banner único** listando todas as variáveis de ambiente que o daemon consulta, marcando cada uma como `[SET]` ou `[MISSING]`. **Sem mostrar valores** — apenas o nome e o estado. Banner sai antes do `server listening` log para que o usuário veja imediatamente o que está configurado.

## Motivação

Hoje o daemon consulta ~9 env vars (`VBM_PORT`, `VBM_BIND`, `VBM_EMBED_URL`, `VBM_EMBED_MODEL`, `VBM_EMBED_API_KEY`, `VBM_EMBED_FORMAT`, `VBM_LLM_MODEL`, `VBM_TTL_DAYS`, `VBM_CORS_ORIGIN`) espalhadas em `server.go`. O usuário precisa ler o código ou a documentação pra saber quais existem e quais ele esqueceu de setar. Erros silenciosos como "rodei sem `VBM_EMBED_API_KEY` e a busca semântica está desligada" só aparecem implicitamente em `slog.Warn` no meio do log de startup. Um banner único e enumerativo elimina esse ruído.

## Comportamento atual

- Variáveis de ambiente são consultadas ad-hoc em `server.go`/`Run()` via `os.Getenv()`.
- Algumas geram warnings ao não estarem setadas (ex.: `VBM_EMBED_URL not set — using stub embedder`).
- Não há lugar único que liste todas. Para descobrir, o usuário precisa `grep -rn "VBM_" daemon/`.

## Comportamento desejado

- Imediatamente após `Run()` ler todas as envs, antes de qualquer outro log de inicialização (embedder/LLM/TTL/CORS), o daemon emite um banner via `slog.Info` com formato:

  ```
  ─── env config ───────────────────────────
    VBM_PORT             [SET]
    VBM_BIND             [MISSING]   (default: 127.0.0.1)
    VBM_EMBED_URL        [SET]
    VBM_EMBED_MODEL      [MISSING]   (default: openai/text-embedding-3-small)
    VBM_EMBED_API_KEY    [SET]
    VBM_EMBED_FORMAT     [MISSING]   (default: ollama)
    VBM_LLM_MODEL        [MISSING]   (default: gpt-4o-mini)
    VBM_TTL_DAYS         [MISSING]   (retention disabled)
    VBM_CORS_ORIGIN      [MISSING]
  ──────────────────────────────────────────
  ```

- Para cada var: nome alinhado em coluna fixa + `[SET]` ou `[MISSING]`. Quando `[MISSING]` e há default ou consequência clara, exibir entre parênteses (ex.: `(default: 127.0.0.1)`, `(retention disabled)`).
- Implementação centralizada num helper interno (`logEnvBanner()`) que recebe uma lista declarativa `[]envSpec{name, defaultDesc}`. Adicionar nova env passa a ser uma linha nessa lista.

## Análise de impacto

### Entidades afetadas
Nenhuma. Mudança restrita a logging de startup.

### Backend
- `daemon/internal/server/server.go` — adicionar função `logEnvBanner()` chamada como primeira coisa em `Run()`, antes de qualquer `os.Getenv()` para inicialização. Na verdade ainda funciona se chamada depois — o banner só lê os envs (não muta nada).
- Pequena refatoração: a lista declarativa de envs vira a fonte de verdade. Os warnings ad-hoc tipo `slog.Warn("VBM_EMBED_URL not set — using stub embedder…")` continuam onde estão (descrevem **consequência funcional**, não estado), pra preservar o sinal específico.

Nenhum impacto.

## Decisões necessárias antes de implementar

1. **Coluna fixa** — proposta: 22 chars (cabe `VBM_EMBED_API_KEY` com folga).
2. **Quando emitir** — proposta: imediatamente no início de `Run()`, antes de `dataDir` e tudo mais. Se houver erro depois (data dir, listen, etc.) o usuário pelo menos viu a config.
3. **Formato** — proposta: usar `slog.Info` com mensagem multi-linha. Alternativa seria `fmt.Println` direto, mas slog mantém consistência com o resto dos logs.

Decisões 1–3 podem ser tomadas durante a implementação se não houver opinião forte.

---

## Critério de aceite

1. `cd daemon && make build && ./bin/vbmd server` (sem nenhuma env setada além do necessário) imprime o banner com a lista de 9 vars, todas marcadas como `[SET]` ou `[MISSING]`, antes de `server listening`.
2. `VBM_EMBED_URL=… VBM_EMBED_API_KEY=… ./bin/vbmd server` mostra essas duas como `[SET]`, demais como `[MISSING]`.
3. **Nenhum valor sensível** aparece no banner (em particular `VBM_EMBED_API_KEY` nunca expõe a chave — apenas o estado).
4. Adicionar uma 10ª var futuramente é uma única linha em `envSpecs` em `server.go`.
5. `go test ./...` continua verde.
6. Nenhuma mudança no comportamento funcional do daemon — só na saída de log.

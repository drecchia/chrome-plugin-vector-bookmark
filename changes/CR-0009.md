# CR-0009 — Aba "Manage": manutenção de tags duplicadas com merge assistido por LLM

**Data:** 2026-06-27
**Solicitante:** Danilo Recchia
**App Version:** daemon 1.1.4 / extension 1.1.4
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Daemon HTTP UI (nova aba), Daemon store (operação de merge), Daemon LLM client (sugestão de merges), proto/types.ts

---

## Descrição da mudança

Nova aba **Manage** na UI HTTP do daemon (`/ui`), ao lado de Search / Tags / Hot Words / Timeline / Exclusions. Resolve a variância de tags (`ml`, `machine-learning`, `machine learning`) que fragmenta o filtro de tags e dificulta a busca manual.

A aba tem duas seções:

1. **Scan assistido por LLM.** Botão "Scan for duplicate tags" → `POST /tags/merge/suggest`. O daemon lista todas as tags (`ListTags`) e pede ao LLM (`SuggestTagMerges`) para agrupar apenas **near-duplicates** (variações de grafia/espaço, singular/plural, abreviação↔expansão, sinônimos óbvios do mesmo conceito) e escolher uma canônica por grupo. Cada grupo vira um card: chips das variantes + `<select>` da canônica (pré-selecionado na escolha do LLM, **sobrescrevível** pelo usuário) + botões **Approve merge** / **Dismiss**. Approve → `POST /tags/merge {from, to}`.

2. **Merge manual.** Dois inputs ("merge __ into __", com `datalist` populado de `/tags`) + botão Merge → mesmo `POST /tags/merge`. Para duplicatas que o scan não pega.

A operação de merge renomeia cada tag `from` para `to` em **todas** as páginas (`page_tags`), com dedup nas páginas que já carregam ambas.

## Motivação

Tags acumulam variância ao longo do tempo. Sem uma operação de rename/merge, a única forma de consolidar era reindexar página a página em set-mode — inviável. O resultado é um filtro de tags poluído (`ml` e `machine-learning` como entradas separadas) e busca manual fragmentada. Esta CR adiciona a primitiva de merge que faltava, com revisão humana obrigatória (nada é mesclado sem Approve), e usa o LLM apenas como sugestão.

## Comportamento atual

- Tags só podem ser definidas por página: `UpdatePageTags` (set-mode) ou `persistTags` (merge/add). **Não existe** rename nem merge entre tags.
- A UI tem 5 abas; nenhuma permite manutenção/consolidação de taxonomia.
- `internal/llm` expõe `Summarize` e `SuggestTags`; nada que opere sobre a lista de tags.

## Comportamento desejado

### Store (`internal/store/sqlite.go`)
- Novo `MergeTags(from []string, to string) (pagesAffected int, err error)`:
  - Normaliza `to` (`normalizeTag`); erro se vazio. Normaliza cada `from`, descarta vazios e os iguais a `to`.
  - Numa única tx, por tag `from`: coleta `page_id`s afetados (para a contagem), `UPDATE OR IGNORE page_tags SET tag=:to, created_at=:now WHERE tag=:from` (o `OR IGNORE` pula colisões com a PK `(page_id, tag)` quando a página já tem `to`), depois `DELETE FROM page_tags WHERE tag=:from` (remove as sobras que o UPDATE pulou).
  - Retorna nº de páginas distintas afetadas.
- Sem migração de schema (`page_tags` já existe; schema permanece em V7).

### LLM (`internal/llm/llm.go`)
- Novos tipos `TagStat{Tag string; Count int}` e `MergeGroup{Canonical string; Variants []string}` (definidos no pacote `llm`, sem importar `store`, para manter o pacote livre de dependências).
- Novo `SuggestTagMerges(ctx, tags []TagStat) ([]MergeGroup, error)`, modelado em `SuggestTags`: cap, system+user, retry-once em 5xx, parser tolerante a fences (`parseMergeGroupsJSON`). Descarta grupos sem variantes.
- Prompt externalizável via `VBM_LLM_PROMPT_TAG_MERGE_FILE` (+ fallback embedded `defaultTagMergePrompt`), usando `loadPromptOrDefault`. Novo campo `tagMergePrompt` no `Client`, carregado em `New()`. O prompt instrui a **só** mesclar near-duplicates e, na dúvida, não mesclar.

### Routes (`internal/server/routes.go`)
- `POST /tags/merge/suggest` → 503 se `llmClient==nil`; senão `ListTags` → `SuggestTagMerges` → `{groups:[{canonical, variants}]}`.
- `POST /tags/merge` — body `{from:[...], to:"..."}` → valida, `MergeTags` → `{tag, pagesAffected}`.

### UI (`uiHTML` em routes.go)
- Botão de aba `Manage` + `<div id="manage-panel">` com as duas seções. Scan é **lazy** (só roda no clique do botão, não na abertura da aba — chamada de LLM é cara), seguindo o padrão de Hot Words/Timeline. `datalist` de tags carregado ao abrir a aba. Render dos grupos e CRUD-then-refresh seguem o padrão da aba Exclusions. 503 → mensagem "LLM not configured".

### Env banner (`internal/server/server.go`)
- `VBM_LLM_PROMPT_TAG_MERGE_FILE` adicionado ao slice `envSpecs`.

### proto/types.ts
- `MergeGroup`, `SuggestTagMergesResponse`, `MergeTagsRequest`, `MergeTagsResponse` (a UI embedada é JS puro e não importa; tipos ficam aqui por convenção).

## Análise de impacto

### Entidades afetadas
Nenhuma mudança de schema. `MergeTags` apenas muta linhas de `page_tags`.

### Consistência de busca
Tags vivem só em `page_tags`; BM25/embeddings/FTS são sobre `chunks`. O filtro `?tag=` restringe candidatos via join em `page_tags`. Logo, um merge **não** exige re-embed nem rebuild de FTS — é uma mutação barata e isolada.

### Conflitos com decisões existentes
- **D-002 (provider OpenAI-compat único):** respeitada — reusa `VBM_EMBED_URL`/`VBM_LLM_MODEL`.
- **D-005 (reuso de taxonomia via prompt):** alinhada — o prompt de merge é conservador.
- **D-007 (prompts externalizáveis):** estendida — novo `VBM_LLM_PROMPT_TAG_MERGE_FILE` segue o mesmo padrão `loadPromptOrDefault`.

### CRs relacionadas
- CR-0003 — `SuggestTags`, padrão seguido por `SuggestTagMerges`.
- CR-0006 / D-007 — externalização de prompt, padrão seguido.

## Critério de aceite

1. `cd daemon && make build && go test ./...` — verdes (inclui novos testes `TestMergeTags_*`).
2. `/ui` mostra a aba **Manage**.
3. Merge manual `ml → machine learning`: `curl -XPOST localhost:7532/tags/merge -d '{"from":["ml"],"to":"machine learning"}'` → `{tag, pagesAffected}`; `/tags` não lista mais `ml`; `/pages?tag=machine learning` inclui as páginas.
4. Página com **ambas** `ml` e `machine-learning` termina com uma única linha `machine-learning` (sem erro de PK).
5. Com `VBM_EMBED_URL` setado: "Scan for duplicate tags" renderiza grupos; trocar a canônica no dropdown; Approve → card some e `/tags` reflete o merge. Sem LLM → 503 e mensagem "LLM not configured".
6. Banner do startup lista `VBM_LLM_PROMPT_TAG_MERGE_FILE`.
7. `npm run typecheck` verde.

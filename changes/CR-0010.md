# CR-0010 — Indexação confiável: fim do falso-sucesso no fluxo popup → SW → daemon

**Data:** 2026-07-07
**Solicitante:** Danilo Recchia
**App Version:** 1.2.0
**Status:** implemented
**Urgência:** high
**Domínios afetados:** Extension (popup, service worker, daemon-client), Daemon (queue, store, API HTTP)

---

## Descrição da mudança

Hoje o feedback de "página indexada" (badge verde + toast "Indexed" + contador) é emitido quando o
daemon responde `202 {"queued":true}` ao `POST /ingest` — ou seja, quando o trabalho apenas **entrou
na fila**, não quando terminou. Se o embed via OpenRouter falhar no worker assíncrono, nada é gravado
e o usuário nunca fica sabendo. Este CR torna o feedback **verídico**: o service worker passa a
confirmar a indexação real (poll em `/page` até `indexed=true` ou falha explícita), o daemon ganha
retry com backoff no embed + estado `failed` persistente e consultável na fila, e o popup ganha
estados visuais honestos (spinner durante, erro com botão Retry, contador atualizado só após
confirmação).

## Motivação

Revisão de UX (2026-07-07) identificou quatro sintomas reportados pelo usuário, todos com causa raiz
no mesmo problema de arquitetura ("202 tratado como sucesso final"):

1. **Tarja verde com indexação falhada** — o badge fica verde no 202; se o embed falhar depois
   (erro OpenRouter), a queue row fica `pending` para sempre e o conteúdo nunca é indexado. O
   usuário confia num estado que é mentira.
2. **Contador não incrementa** — o popup refaz `/status` 300ms após o `ingest_complete`
   (`App.tsx:148`), mas o embed real leva segundos; a leitura quase sempre chega antes do write.
3. **Erro de OpenRouter sem retry** — falha de embed deixa a row `pending` e só é retentada no
   **restart do daemon** (`server.go:135`). Nenhum retry em runtime, nenhum feedback, nenhuma ação
   manual possível.
4. **Feedback visual fraco** — Confirm não desabilita (double-click = double ingest), toast
   "Indexing…" é substituído no 202 (não na conclusão), toast some em 2.5s, e reabrir o popup
   perde qualquer noção de ingest em andamento.

Bugs laterais encontrados na mesma revisão e cobertos aqui:

5. **Drop silencioso com 202** — `Queue.Enqueue` descarta a request se o buffer (cap 256) estiver
   cheio (`queue.go:69-75`), mas o handler já respondeu `{"queued":true}`.
6. **Requeue no restart perde tags** — a tabela `queue` não persiste `tags`/`set_tags`; o
   re-enqueue de pendentes no boot (`server.go:135` → `LoadPendingItems`) reconstrói o
   `IngestRequest` sem as tags que o usuário digitou.
7. **Badge de erro enganoso** — falha no `/ingest` (ex.: 502 do `llm_summary`) seta o badge
   vermelho `disconnected` ("daemon unreachable") mesmo com o daemon up (`service-worker.ts:371`).

## Comportamento atual

Fluxo (referências no código em v1.2.0):

```
popup Confirm → popup_force_index → SW pendingIngest.set(tabId)
  → force_extract → content script → page_viewed
  → SW handlePageViewed → POST /ingest
       daemon: (llm_summary? Summarize síncrono) → q.Enqueue + AddQueueItem → 202
  → SW: badge 'indexed' (verde) + broadcast ingest_complete{ok:true}   ← AQUI ESTÁ O FALSO-SUCESSO
  → popup: toast "Indexed", refetch /status após 300ms
       daemon (async): embedWorker chunk+embed → writer → INSERT pages(indexed=1) + DELETE queue row
       em erro de embed: row fica status='pending' eterno; log e nada mais
```

- `queue` (SQLite): `id, url, title, text, visit_ts, dwell_ms, domain, status('pending'), created_at, updated_at` — sem tags, sem contagem de tentativas, sem erro.
- `GET /page?url=` retorna apenas `{exists, indexed}`.
- `GET /status` retorna `{visited, indexed, pending, ...}` — sem noção de falha.
- Badge states: `tracking(0) < disconnected(1) < blocked(2) < visited(3) < indexed(4)`.
- Popup fechado após Confirm: o fluxo continua no SW (ok), mas o resultado real continua invisível.

## Comportamento desejado

```
popup Confirm (botão desabilita, spinner "Indexing…")
  → SW → /ingest → 202 → badge 'indexing' (novo estado)
  → SW poll GET /page?url= (1s, backoff até 3s, timeout 45s):
       indexed=true            → badge 'indexed' + ingest_complete{ok:true} → popup "Indexed ✓" + refetch /status
       queueStatus='failed'    → badge 'error' + ingest_complete{ok:false, error, retriable:true}
                                  → popup toast de erro persistente + botão "Retry"
       timeout                 → ingest_complete{ok:false, error:'timeout', retriable:true} (badge mantém 'indexing')
  daemon: embed com retry interno (3 tentativas, backoff 1s/4s/10s);
          falha definitiva → queue row status='failed', last_error, attempts;
          POST /queue/retry {url} re-enfileira a row failed (reset p/ pending)
```

- `202` deixa de significar sucesso em **qualquer** camada de UI.
- Buffer cheio → `/ingest` responde `503 {"error":"ingest queue full"}` (sem falso `queued:true`).
- Reabrir o popup durante um ingest em andamento restaura o estado "Indexing…" (consulta ao SW).
- `Falha de LLM summary` (síncrona, 502) mostra erro real no popup com Retry — badge **não** vira
  `disconnected`.

## Análise de impacto

### Entidades afetadas

- **`queue`** (SQLite) — migração **V9** (V8 já existe: coluna `source`): novas colunas
  `tags_json TEXT NOT NULL DEFAULT '[]'`, `set_tags INTEGER NOT NULL DEFAULT 0`,
  `attempts INTEGER NOT NULL DEFAULT 0`, `last_error TEXT NOT NULL DEFAULT ''`.
  Valores de `status` passam a ser `'pending' | 'failed'` (hoje só `'pending'`; sucesso continua
  sendo DELETE da row).
- **`pages`** — sem mudança de schema.

### Arquivos afetados

**Daemon (Go):**
- `daemon/internal/store/sqlite.go` — migração V9; `AddQueueItem` persiste tags/set_tags;
  `LoadPendingItems` retorna tags/set_tags; novos métodos `MarkQueueItemFailed(url, errMsg, attempts)`,
  `GetQueueItemStatus(url) (status, lastError, ok)`, `RetryQueueItem(url) (found bool)`;
  `GetStatus` ganha contagem `failed`.
- `daemon/internal/queue/queue.go` — `Enqueue` retorna `bool` (falso = buffer cheio); retry com
  backoff exponencial no `embedWorker` (3 tentativas por request: 1s, 4s, 10s); falha definitiva
  chama `MarkQueueItemFailed` em vez de abandonar a row como `pending`.
- `daemon/internal/server/routes.go` — `/ingest` responde 503 quando `Enqueue` retorna falso e não
  persiste a queue row; `GET /page` estende resposta com `queueStatus` + `lastError`; novo
  `POST /queue/retry` `{url}`; `/status` inclui `failed`.
- `daemon/internal/server/server.go` — boot requeue (`LoadPendingItems`) preserva tags; rows
  `failed` **não** são re-enfileiradas automaticamente no boot (só via `/queue/retry`).
- `daemon/internal/llm/` — (opcional, escopo incluído) retry 2x com backoff para respostas
  429/5xx transitórias do provider em `Summarize`.
- `daemon/internal/store/sqlite_test.go` e/ou novos testes — cobertura dos métodos novos.

**Extensão (TypeScript):**
- `proto/types.ts` — `PageStatusResponse` ganha `queueStatus?: 'pending' | 'failed'` e
  `lastError?: string`; `StatusResponse` ganha `failed?: number`; novo tipo p/ retry.
- `extension/src/background/daemon-client.ts` — `retryQueueItem(url)`; `pageStatus` já
  desserializa os campos novos.
- `extension/src/background/service-worker.ts` — novo badge state `indexing` (prioridade entre
  `visited` e `indexed`) e `error` (cor/título próprios, distinto de `disconnected`); loop de
  confirmação pós-202 (`pollIngestOutcome(url, tabId)`: intervalo 1s com backoff até 3s, timeout
  45s); mapa `inFlightIngests: Map<string /*url*/, ...>` + handler `popup_ingest_state` para o
  popup restaurar estado; handler `popup_retry_ingest`; broadcast `ingest_complete` só após
  desfecho real, com `{ok, error?, retriable?}`; falha síncrona do `/ingest` (4xx/5xx) usa badge
  `error`, não `disconnected`.
- `extension/src/popup/App.tsx` — Confirm desabilitado + spinner enquanto in-flight; toast
  "Indexing…" persistente até o desfecho; toast de erro **persistente** (sem auto-dismiss) com
  botão "Retry"; refetch de `/status` disparado pelo `ingest_complete{ok:true}` (remove o
  `setTimeout(300)`); no mount, envia `popup_ingest_state` para a URL ativa e restaura
  "Indexing…" se houver ingest em andamento.

### Backend

- **Migração V9** (aditiva, sem conversão de dados — colunas com DEFAULT).
- **`POST /ingest`**: passa a poder responder `503 {"error":"ingest queue full"}`. Contrato do 202
  inalterado na forma, mas a semântica documentada muda para "aceito para processamento".
- **`GET /page?url=`**: resposta estendida (retrocompatível — campos novos opcionais).
- **`POST /queue/retry`** (novo): body `{"url": "..."}`; `200 {"retried":true}` se havia row
  `failed`; `404 {"error":"no failed item for url"}` caso contrário. Coberto pelo auth token
  (CR-0008) como os demais endpoints.
- **`GET /status`**: campo novo `failed` (int).
- **Worker de embed**: retry interno segura o slot do worker durante o backoff (aceitável: 4
  workers, daemon local, backoff total ~15s).

### Frontend / Mobile

- Popup: estados novos (in-flight, erro persistente com Retry, restauração ao reabrir).
- Badge: dois estados novos — `indexing` (ex.: roxo `#8b5cf6`, "indexing…") e `error`
  (ex.: laranja `#f97316`, "indexing failed — open popup to retry"). Tabela de prioridade vira:
  `tracking(0) < disconnected(1) < blocked(2) < visited(3) < indexing(4) < error(5) < indexed(6)`
  — `error` acima de `indexing` para o desfecho vencer, `indexed` no topo como hoje.
- UI HTTP do daemon (`/ui`): fora de escopo (nenhuma mudança obrigatória; o campo `failed` do
  `/status` pode ser exibido futuramente).

### Dados existentes

Não requer migração de dados — colunas novas com DEFAULT cobrem rows existentes. Rows `pending`
antigas (fossilizadas por falhas de embed pré-CR) passam a ser re-enfileiradas no próximo boot já
com o retry novo; se falharem de novo, viram `failed` e ficam visíveis/acionáveis.

### Integrações externas

OpenRouter (via `VBM_EMBED_URL`) — o retry com backoff reduz falhas transitórias (429/5xx). Nenhum
contrato externo muda.

### Conflitos com decisões existentes

Nenhum conflito identificado. D-002 (provider único OpenAI-compat) e D-003 (modos de ingest como
enum) permanecem intactos. A semântica do 202 muda, mas nenhuma decisão D-NNN a fixava.

### CRs relacionadas

- **CR-0001** — criou o painel de indexação do popup cujo feedback este CR corrige.
- **CR-0006** — fluxo `manual`/intents que compartilham o mesmo caminho `page_viewed → /ingest`
  (beneficiados automaticamente pelo poll de confirmação).
- **CR-0008** — o endpoint novo `/queue/retry` deve respeitar o mesmo middleware de auth.

## Critério de aceite

Cenários verificáveis (com daemon local + extensão carregada):

1. **Sucesso real**: com `VBM_EMBED_API_KEY` válida, Confirm em uma página → botão desabilita,
   toast "Indexing…" persiste, badge fica roxo (`indexing`) e só então verde; toast vira
   "Indexed"; contadores `visited`/`indexed` do popup refletem o incremento **sem** reabrir o
   popup. `GET /page?url=` retorna `indexed=true`.
2. **Falha de embed com retry**: com `VBM_EMBED_API_KEY` inválida (simula OpenRouter fora),
   Confirm → daemon loga 3 tentativas com backoff; queue row termina `status='failed'` com
   `last_error` preenchido; badge fica laranja (`error`); popup mostra toast de erro persistente
   com botão "Retry"; contador **não** incrementa; toast **não** diz "Indexed" em momento algum.
3. **Retry manual**: restaurar a API key, clicar "Retry" no popup → `POST /queue/retry` re-enfileira,
   poll reinicia, fluxo termina como no cenário 1. `GET /status` mostra `failed` decrementado.
4. **Popup fechado mid-flight**: Confirm e fechar o popup imediatamente → badge progride
   roxo→verde sozinho; reabrir o popup durante o processamento mostra "Indexing…" restaurado;
   reabrir depois mostra "✓ This page is indexed".
5. **Falha síncrona (llm_summary)**: modo "Summarize via LLM" com LLM indisponível → popup mostra
   o erro real (502 "LLM call failed") com Retry; badge fica laranja (`error`), **não** vermelho
   `disconnected`; `/healthz` continua ok.
6. **Buffer cheio**: `POST /ingest` com fila lotada responde `503 {"error":"ingest queue full"}`
   (teste Go unitário com queue de cap 1 bloqueada) e não cria queue row.
7. **Requeue com tags**: enfileirar ingest com tags, derrubar o daemon antes do embed concluir,
   subir de novo → página indexada **com as tags originais** (`GET /page` + `page_tags`).
8. **Double-click**: dois cliques rápidos em Confirm geram no máximo 1 `/ingest`.
9. **Regressão**: `cd daemon && make test` verde; `cd extension && npm run typecheck` limpo;
   fluxo passivo de dwell (badge azul `visited`) inalterado.

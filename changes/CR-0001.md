# CR-0001 — Painel unificado de indexação com tags + modo de extração

**Data:** 2026-05-02
**Solicitante:** Danilo Recchia
**App Version:** extension 0.1.0
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Extension popup (UX), Daemon HTTP API, Daemon store (tags), Daemon LLM integration (novo)

---

## Descrição da mudança

Substituir os dois botões atuais do popup ("Index this page now" e "Index with tag") por **um único botão "Index this site now"** que, ao ser clicado, expande um painel inline com:

1. Campo de **tags separadas por vírgula** (n tags, livres). Quando a página já está indexada, o campo vem **pré-preenchido com as tags existentes** carregadas do banco e é **editável** (remover/adicionar).
2. Seletor de **modo de indexação** com quatro opções:
   - **Indexar todos os chunks** — full-text Readability (comportamento atual).
   - **Resumo via LLM** — daemon gera um resumo do conteúdo via LLM e indexa apenas o resumo.
   - **Texto manual** — usuário cola/digita o texto a indexar; substitui a extração automática.
   - **Apenas meta tags + título** — indexa só `description/keywords/og:*/author + title` (sem corpo).
3. Botão de confirmação que aplica tags + modo escolhido em uma única chamada `/ingest`.

## Motivação

A iteração anterior (CR não numerada, já mergeada) introduziu um segundo botão "Index with tag" e um input de tag única. O usuário avalia agora que:

- Dois botões competem por atenção e duplicam intent (indexar é sempre o objetivo; tag é parametrização).
- Faltava controle sobre **como** o conteúdo entra no índice — full-text é caro pra páginas longas, ruidoso pra páginas com muito boilerplate, e às vezes o usuário quer apenas um resumo curto pra recuperação semântica.
- Faltava forma de editar tags já atribuídas sem precisar "esquecer" e re-indexar.

A nova proposta unifica os botões e expõe os modos de extração, dando ao usuário uma ferramenta única e mais poderosa.

## Comportamento atual

- Popup tem dois botões: **"Index this page now"** (full-text, sem tag) e **"Index with tag"** (full-text + uma única tag).
- Tags são acumulativas — `INSERT OR IGNORE` no `page_tags`. Não há remoção de tag pelo popup.
- Sempre extrai via Readability + chunking (512 tok / overlap 64).
- Não existe integração com LLM para resumo.
- Não existe modo de texto manual.
- `GET /page?url=…` retorna apenas `{exists, indexed}` — não expõe tags.

## Comportamento desejado

### Popup
- **Apenas um botão "Index this site now"**.
- Click expande painel inline com:
  - Input de tags (CSV, separadas por vírgula). Pré-preenchido com tags atuais quando a página já está indexada.
  - Radio com 4 modos: `full_text` (default) | `llm_summary` | `manual` | `meta_only`.
  - Quando `manual` está selecionado: revela `<textarea>` para o usuário colar/digitar o texto.
  - Botão "Confirm" único.
- Após confirmar: ingest é disparado, tags são sincronizadas (set-mode), badge fica verde.
- Lista de tags conhecidas (autocomplete) continua disponível via datalist.

### Daemon
- `POST /ingest` ganha campo opcional `mode: "full_text" | "llm_summary" | "manual" | "meta_only"` (default `full_text` — backwards compatible).
- Quando `mode == "llm_summary"`: daemon chama o LLM (OpenAI-compat, mesmo `VBM_EMBED_API_KEY` + endpoint, novo env `VBM_LLM_MODEL`, default `gpt-4o-mini`) com prompt do tipo "summarize for retrieval"; o resumo retornado substitui o `text` antes do chunking.
- Quando `mode == "manual"`: o `text` enviado pelo client é usado direto, sem extração no content script (popup envia o texto que o usuário colou).
- Quando `mode == "meta_only"`: ingest reusa o pipeline do `/visit` (título + meta tags concatenados, sem corpo).
- Tags em ingest passam a operar em **set-mode** (substituição), não accumulate-mode: a lista enviada se torna a lista final daquela página. Vazio = remove todas as tags.
- `GET /page?url=…` retorna também `tags: string[]` da página existente.

## Análise de impacto

### Entidades afetadas

- **`pages`** — sem mudança de schema.
- **`page_tags`** — sem mudança de schema. Mas semântica de escrita muda: ingest passa a fazer DELETE+INSERT atômico (set-mode), em vez de apenas INSERT OR IGNORE.
- Novo: pacote `internal/llm` (cliente OpenAI-compat para chat completions).

### Backend

- `daemon/internal/store/sqlite.go`:
  - `IngestRequest` ganha campo `Mode string`.
  - `Ingest()`: nova lógica para diff de tags (set-mode) — `DELETE FROM page_tags WHERE page_id=? AND tag NOT IN (...)` + `INSERT OR IGNORE` para as novas. Tudo dentro da tx existente.
  - Helper `replacePageTags(tx, pageID, tags []string, now int64)`.
- `daemon/internal/server/routes.go`:
  - `ingestRequest` struct: novo campo `Mode string`.
  - Handler de `/ingest`:
    - `mode == "llm_summary"`: chamar `llm.Summarize(ir.Text)` antes de enfileirar.
    - `mode == "manual"`: usar `ir.Text` como veio.
    - `mode == "meta_only"`: pipeline equivalente ao `/visit` body-builder (título + meta).
    - `mode == "full_text"` (ou vazio): comportamento atual.
  - `GET /page` passa a incluir `tags []string`. Adicionar `Store.GetPageTags(url string)`.
- `daemon/internal/llm/llm.go` (NOVO):
  - Reusa env `VBM_EMBED_API_KEY` + endpoint base (mesmo padrão OpenAI-compat já em uso).
  - Novo env `VBM_LLM_MODEL` (default `gpt-4o-mini`).
  - Função `Summarize(ctx, text string) (string, error)` com prompt fixo: "Summarize the following web page content for retrieval/search purposes. 200-400 words. Preserve technical terms, names, dates, key claims. Output plain prose, no headers."
  - Timeout 30s. Retry uma vez em 5xx/timeout.
  - Erro propagado como 502 ao client (com mensagem clara para flash no popup).
- Sem migração de schema necessária.

### Frontend / Mobile (popup da extensão)

- `extension/src/popup/App.tsx`:
  - Remover botão "Index with tag" e seu handler `handleIndexWithTag`.
  - Renomear botão restante para **"Index this site now"** e fazer com que o click apenas alterne o painel (não dispara ingest direto).
  - Novo estado: `panelOpen: boolean`, `tagsCSV: string`, `mode: 'full_text'|'llm_summary'|'manual'|'meta_only'`, `manualText: string`.
  - Ao abrir o painel: chamar `popup_get_page_tags` (novo handler no SW, via `pageStatus` estendido) para pré-preencher `tagsCSV`.
  - Painel:
    - input CSV com datalist `vbm-known-tags`.
    - 4 radios para o modo.
    - textarea visível só quando `mode === 'manual'`.
    - botão "Confirm".
  - Submit chama `popup_force_index` com `{ tags: parseCSV(tagsCSV), mode, manualText? }`.
- `extension/src/background/service-worker.ts`:
  - `pendingTags` map continua, mas armazena também `mode` e `manualText` opcional.
  - Para `mode === 'manual'`: pula o `force_extract` no content script — envia direto `/ingest` com o texto manual.
  - Para `mode === 'meta_only'`: ainda precisa do content script (pra coletar meta tags atuais) — content script já entrega `meta` no `page_viewed`; o body do ingest fica só com `metaParts`, sem `msg.text`.
  - Para `llm_summary` e `full_text`: pipeline atual (Readability via content script). Modo é repassado ao daemon via campo `mode` do ingest.
- `extension/src/background/daemon-client.ts`:
  - `ingest()` passa `mode` quando definido.
  - `pageStatus()` retorna também `tags`.
- `proto/types.ts`:
  - `IngestRequest.mode?: 'full_text' | 'llm_summary' | 'manual' | 'meta_only'`.
  - `PageStatusResponse.tags?: string[]`.

### Dados existentes

Não requer migração de banco. A mudança de semântica de tags (accumulate → set) afeta apenas operações futuras. Tags já gravadas continuam intactas.

### Integrações externas

- Nova dependência: provider OpenAI-compat (mesmo já usado por embeddings) precisa suportar chat completions. Se o usuário rodar com stub embedder e sem `VBM_EMBED_API_KEY`, o modo `llm_summary` retorna 503 com mensagem "LLM not configured". Os outros 3 modos seguem funcionando.

### Conflitos com decisões existentes

- **CLAUDE.md / regras de negócio §4**: "Dedup de página: revisitas atualizam a página existente." — mantido. Set-mode de tags é coerente com revisita atualizando.
- **CLAUDE.md / `chunks_fts` rebuild síncrono**: o DELETE de tags não toca em chunks/FTS, então a regra não é violada.
- Nenhum conflito real identificado.

Nenhum CR anterior — esta é a CR-0001.

## Decisões necessárias antes de implementar

1. **Prompt do summarizer** — proposto: "Summarize the following web page content for retrieval/search purposes. 200-400 words. Preserve technical terms, names, dates, key claims. Output plain prose, no headers." Ajustar se quiser idioma específico ou tom diferente.
2. **Modelo default** — proposto: `gpt-4o-mini` (rápido, barato, ~150ms p50). Trocar via `VBM_LLM_MODEL` se preferir Llama/Mistral via Ollama OpenAI-compat.
3. **Limite de tamanho de input do LLM** — proposto: truncar `text` a 32k chars antes de enviar (proteção contra páginas gigantes que estouram a janela do modelo). Ajustar se houver preferência.

Decisões 1–3 podem ser tomadas durante a implementação se o usuário não tiver opinião forte.

## Critério de aceite

1. Popup mostra **apenas um botão** "Index this site now". Antiga UI com input + botão "Index with tag" não aparece mais.
2. Click no botão expande painel inline com input CSV + 4 radios + (condicional) textarea + botão "Confirm".
3. Ao abrir o painel em página já indexada com tags `[a, b]`, o campo vem com `"a, b"`. Remover `b`, salvar e reabrir → vem só `"a"` (set-mode confirmado via `sqlite3 vbm.db "SELECT tag FROM page_tags WHERE page_id=…"`).
4. Modo `full_text` em uma página de blog produz N chunks no `chunks` table (igual hoje).
5. Modo `llm_summary` em uma página com `VBM_EMBED_API_KEY` configurado produz **1 chunk** com texto curto (~200-400 palavras) que NÃO é o body da página.
6. Modo `manual` ignora o conteúdo da página: indexa apenas o que foi colado no textarea (`SELECT text FROM chunks` traz exatamente o texto digitado).
7. Modo `meta_only` produz 1 chunk pequeno só com título + descrição/keywords/og.
8. Modo `llm_summary` sem `VBM_EMBED_API_KEY` → flash vermelho "LLM not configured".
9. Tags vazias no submit removem todas as tags da página (`SELECT COUNT(*) FROM page_tags WHERE page_id=…` → 0).
10. `curl /tags` reflete o estado correto após adições/remoções via popup.
11. `/ui` aba "Tags" e dropdown de busca continuam funcionando após CR.
12. `go test ./...` verde, `npm run typecheck` verde, `npm run build` verde.

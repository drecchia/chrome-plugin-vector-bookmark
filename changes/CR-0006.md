# CR-0006 — Modo manual com título+meta implícitos + prompts externalizáveis

**Data:** 2026-05-02
**Solicitante:** Danilo Recchia
**App Version:** daemon 0.1.0 / extension 0.1.0
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Extension popup, Extension content script, Daemon LLM client (configuração de prompts)

---

## Descrição da mudança

Duas mudanças complementares numa só CR (escopo coeso, ambas tocam fluxo manual + LLM):

1. **Modo `manual` passa a incluir título + meta tags da página implicitamente.** Hoje o popup pula o content script no `manual`, então só o texto digitado é indexado — o título e meta da página real não aparecem na busca FTS. A correção faz o SW disparar `force_extract` com novo intent `manual`, que extrai apenas título+meta (sem Readability) e emite `page_viewed` normalmente. O `handlePageViewed` é estendido para, quando há `manualText` pendente, concatená-lo ao bloco de meta antes do `/ingest`. Mantém-se uma única origem de orquestração de ingest (handlePageViewed), simétrico com os outros modos.

2. **Prompts do `Summarize` e `SuggestTags` ficam externalizáveis** via dois env vars que apontam para arquivos **Markdown** em disco:
   - `VBM_LLM_PROMPT_SUMMARIZE_FILE` (ex.: `summarize.md`)
   - `VBM_LLM_PROMPT_SUGGEST_TAGS_FILE` (ex.: `suggest_tags.md`)

   Quando setados e o arquivo é legível, o conteúdo substitui o prompt padrão. Quando ausentes, vazios, ou o arquivo está inválido, **fallback para o prompt hardcoded atual** (com `slog.Warn` no fallback explicando o motivo). Lido **uma vez** no startup; trocar prompt = restart.

   Markdown puro como conteúdo (com headers, listas, code fences se útil) — daemon trata o arquivo inteiro como texto e envia o conteúdo como system prompt do LLM. Os modelos modernos lidam bem com markdown na system message; nada precisa ser strippado.

## Motivação

**Parte 1 — manual com meta**: hoje há uma inconsistência sutil. Os modos `full_text`, `selection`, `meta_only`, `yt_*` todos incluem título+meta no chunk indexado. **Apenas `manual` não inclui.** Resultado: usuário cola texto de uma página titulada "Best chocolate cake recipes" e depois busca "chocolate cake" → não encontra. É um foot-gun silencioso.

**Parte 2 — prompts externos**: hoje os prompts vivem hardcoded em `internal/llm/llm.go`. Toda mudança de prompt (afinar resumo, mudar idioma, ajustar instrução de tags) exige rebuild + restart. Externalizando:
- Permite iteração rápida sem recompilar.
- Permite o usuário ajustar idioma/estilo sem mexer no código.
- Mantém os defaults sensatos via fallback embedded — quem não setar nada continua funcionando.

## Comportamento atual

### Manual mode
- Popup envia `popup_force_index` com `{mode: 'manual', manualText: '...'}`.
- SW detecta `mode === 'manual' && !intent` → **pula content script**, chama `/ingest` direto com `text: manualText`.
- Resultado: chunk indexado = só o que o usuário digitou. Título da página vai pra `pages.title` (display only, não FTS). Meta tags somem.

### Prompts
- `daemon/internal/llm/llm.go`:
  - `summarizePrompt` é uma constante `const summarizePrompt = "..."`.
  - Em `SuggestTags`, a system message é uma string inline (`system := "You generate tags..."`) construída a cada chamada.
- Sem hook de override.

## Comportamento desejado

### Manual mode

**Nota sobre `page_viewed`:** nesse codebase, `page_viewed` é a **mensagem que dispara a indexação** (não um conceito separado de "tracking"). O fluxo padrão de indexação é: content script extrai → emite `page_viewed` → SW `handlePageViewed` chama `ingest()`. Toda mudança nesse contrato deve preservar essa semântica — incluindo o modo manual.

- Popup envia mesmo payload (`popup_force_index` com `{mode: 'manual', manualText}`).
- SW armazena `pendingIngest = { mode: 'manual', tags, manualText }` (campo `manualText` adicionado ao tipo `PendingIngest`).
- SW dispara `force_extract` com novo intent **`manual`**.
- Content script: branch `'manual'` extrai **apenas título + meta** (sem Readability). Constrói `metaText = title + description + keywords + og + author`. Emite `page_viewed` com `{ url, title, text: metaText, dwellMs: 0, meta }` — exatamente como os outros intents.
- SW `handlePageViewed` reconhece `pending.manualText` e monta:
  ```
  text = metaText ? `${metaText}\n\n${pending.manualText}` : pending.manualText
  ```
  Caso contrário (sem `manualText`), comportamento atual preservado para os demais modos.
- Chama `/ingest` com `mode='manual'`. Resultado: chunk indexado tem título, meta block, e o texto digitado pelo usuário, na mesma ordem dos outros modos.

### Prompts
- Novas envs no banner (CR-0004):
  - `VBM_LLM_PROMPT_SUMMARIZE_FILE` — caminho absoluto para arquivo .txt
  - `VBM_LLM_PROMPT_SUGGEST_TAGS_FILE` — idem
- `internal/llm/llm.go`:
  - Funções de carga: `loadPromptOrDefault(envVar string, fallback string) string`. Lê arquivo se env setada e legível; senão `slog.Warn` (com motivo) e retorna fallback.
  - Carregadas **uma vez** na construção do `Client` (`New()` em `llm.go`), cacheadas em campos `c.summarizePrompt` e `c.suggestTagsPrompt`.
  - `Summarize()` e `SuggestTags()` passam a usar `c.summarizePrompt` / `c.suggestTagsPrompt` em vez das constantes.
- Defaults atuais permanecem como fallback embedded (preservados verbatim — não muda o comportamento de quem não configurar).

## Análise de impacto

### Entidades afetadas
Nenhuma. Sem mudança de schema.

### Backend (Go)
- `daemon/internal/llm/llm.go`:
  - Constants `summarizePrompt` e a string inline em `SuggestTags` viram defaults exportáveis (`defaultSummarizePrompt`, `defaultSuggestTagsPrompt`).
  - Adicionar `summarizePrompt`, `suggestTagsPrompt` como campos do `Client`.
  - Helper `func loadPromptOrDefault(envVar, fallback string) string` (novo, dentro do mesmo arquivo). `os.ReadFile`, `strings.TrimSpace`, fallback em qualquer erro com `slog.Warn`.
  - `New()` invoca o helper duas vezes e seta os campos.
- `daemon/internal/server/server.go`: adicionar `VBM_LLM_PROMPT_SUMMARIZE_FILE` e `VBM_LLM_PROMPT_SUGGEST_TAGS_FILE` ao slice `envSpecs` (CR-0004).

### Extensão (TypeScript)
- `extension/src/content/extract.ts`:
  - Novo intent `manual` no listener `force_extract`. Função `extractMetaOnly()` (interno) constrói o bloco título+meta e emite `page_viewed` com `text = metaText`. Retorno via `sendResponse({ ok: true })` mantido para o callback do SW saber que o content script foi alcançado.
- `extension/src/background/service-worker.ts`:
  - Branch `mode === 'manual' && !intent`: deixar de bypass-ar o content script. Em vez disso, setar `pendingIngest.set(tabId, { mode: 'manual', tags, manualText })` e disparar `force_extract({ intent: 'manual' })`.
  - `handlePageViewed`: estender para reconhecer `pending.manualText`. Quando setado, `text = metaText ? \`${metaText}\\n\\n${pending.manualText}\` : pending.manualText`.
  - Tipo `PendingIngest` ganha campo opcional `manualText?: string`.
- `extension/proto/types.ts`: estender `ExtractIntent` para incluir `'manual'`.
- `extension/src/popup/App.tsx`: **sem mudança visível**. Continua mandando `{mode: 'manual', manualText}`.

### Dados existentes
Não requer migração. Páginas indexadas anteriormente em `manual` continuam como estão; novas indexações em `manual` passam a incluir título+meta.

### Integrações externas
Nenhuma alteração de integração. Os arquivos de prompt são lidos do filesystem local do daemon (path absoluto definido pelo usuário). Não chama nada externo novo.

## Decisões necessárias antes de implementar

1. **Formato do arquivo de prompt** — proposto: **Markdown UTF-8**, conteúdo inteiro do arquivo vira o prompt (com `TrimSpace`). Sem header, sem frontmatter, sem template engine. Daemon trata o conteúdo como texto bruto.
2. **Reload em runtime?** — proposto: **não**. Lido uma vez no startup. Mudou prompt = restart o daemon. Simples e suficiente pra POC.
3. **Caminho recomendado para os arquivos** — proposto: `$VBM_DATA_DIR/prompts/summarize.md` e `$VBM_DATA_DIR/prompts/suggest_tags.md` (ou onde o usuário quiser). Não criar automaticamente — só ler se setado.
4. **Tamanho máximo do arquivo de prompt** — proposto: 16 KB. Acima disso, fallback + warn.

Decisões 1–4 podem ser tomadas durante a implementação se não houver opinião forte.

## Critério de aceite

### Manual + meta
1. Antes da CR: indexar via popup com `manual` colando "qualquer texto" numa página com título "Best chocolate cake recipes" e meta description "10 easy recipes". Buscar `chocolate cake` → não retorna a página.
2. Depois da CR: mesmo cenário → busca retorna a página. Verificar via `sqlite3 vbm.db "SELECT text FROM chunks WHERE page_id=…"` que o chunk começa com `Best chocolate cake recipes\ndescription: 10 easy recipes\n\n[manualText]`.
3. Modos `full_text`, `selection`, `meta_only`, `llm_summary`, `yt_*` continuam funcionando idênticos (regression check).

### Prompts externos
4. Sem `VBM_LLM_PROMPT_*_FILE` setado: comportamento idêntico ao atual (resumo e tags geram normalmente, prompts internos preservados).
5. Com `VBM_LLM_PROMPT_SUMMARIZE_FILE=/tmp/summarize.md` apontando para arquivo Markdown com prompt "Resuma em português, em 100 palavras." → próximo `mode=llm_summary` produz resumo em português curto.
6. Com env apontando pra arquivo inexistente: log `slog.Warn` no startup explicando "could not read file …, using default prompt", funcionalidade segue normal com prompt padrão.
7. Banner do startup (CR-0004) lista as 2 novas envs como `[SET]` ou `[MISSING]` apropriadamente.
8. `go test ./...`, `npm run typecheck`, `npm run build` continuam verdes.

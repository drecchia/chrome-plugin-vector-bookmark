# CR-0005 — Daemon containerizado (Docker)

**Data:** 2026-05-02
**Solicitante:** Danilo Recchia
**App Version:** daemon 0.1.0
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Daemon (build, distribuição), DX (operação)

---

## Descrição da mudança

Adicionar suporte a Docker para o daemon. Concretamente:

1. **Dockerfile multi-stage** em `daemon/Dockerfile`:
   - Stage 1 (`golang:1.22-bookworm`) compila o binário `vbmd` com `CGO_ENABLED=0` (já é Go puro — `modernc.org/sqlite`).
   - Stage 2 (`gcr.io/distroless/static-debian12`) recebe só o binário; entrypoint = `/vbmd server`.
   - Imagem final ~10MB, sem shell, usuário não-root (`nonroot:nonroot`).
2. **Nova env `VBM_DATA_DIR`** no daemon — sobrescreve `nm.DataDir()` quando setada. Sem ela, o comportamento atual (resolve via `$HOME`) é preservado. Necessário porque distroless não tem `$HOME` por padrão e o usuário precisa montar um volume em caminho conhecido (`/data`).
3. `.dockerignore` minimizado para o build.
4. Bind: usuário roda com `VBM_BIND=0.0.0.0` dentro do container e mapeia `-p 127.0.0.1:7532:7532` no host. Já há permissão explícita pra isso em CLAUDE.md ("Use `VBM_BIND` to override interface (Docker only)") e no código (`server.go` faz bind no `VBM_BIND` quando setado).

Sem docker-compose, sem mudança em Makefile.

## Motivação

- Permitir rodar o daemon em qualquer host sem precisar instalar Go/toolchain — um `docker run` basta.
- Simplifica deploy em homelab/NAS (Synology, Unraid, Proxmox VMs).
- Padroniza diretório de dados: o usuário não precisa entender resolução de `~/.local/share/vbm` em cada SO; basta montar volume em `/data`.
- Build reprodutível sem depender da versão de Go local do desenvolvedor.

## Comportamento atual

- Daemon distribuído como binário nativo via `make build` ou `make install` (systemd).
- Diretório de dados resolvido por `nm.DataDir()` lendo `$HOME` (Linux/Mac) ou `%APPDATA%` (Windows).
- Não há imagem Docker.

## Comportamento desejado

- `cd daemon && docker build -t vbmd:dev .` produz imagem usável.
- `docker run --rm -p 127.0.0.1:7532:7532 -v vbm-data:/data -e VBM_BIND=0.0.0.0 -e VBM_DATA_DIR=/data -e VBM_EMBED_URL=… -e VBM_EMBED_API_KEY=… vbmd:dev` sobe o daemon.
- Banner de env vars (CR-0004) mostra `VBM_DATA_DIR` na lista.
- Dados persistem entre `docker run` graças ao volume.
- Extensão Chrome continua chamando `http://127.0.0.1:7532` — transparente, porque o port mapping está em loopback do host.

## Análise de impacto

### Entidades afetadas
Nenhuma. Sem schema novo; muda apenas onde o arquivo `vbm.db` mora.

### Backend (Go)
- `daemon/internal/nm/nm.go` — `DataDir()` ganha checagem inicial: `if v := os.Getenv("VBM_DATA_DIR"); v != "" { return v, nil }`.
- `daemon/internal/server/server.go` — adicionar `VBM_DATA_DIR` ao `envSpecs` (CR-0004).
- Nenhuma outra mudança de runtime — Go puro, build estático já funciona.

### Frontend / Mobile
Nenhuma mudança. Extensão segue conectando em `127.0.0.1:7532`.

### Build / Distribuição
- Novo arquivo `daemon/Dockerfile` (multi-stage).
- Novo arquivo `daemon/.dockerignore` (exclui `bin/`, `*.db`, etc.).

### Dados existentes
Migração entre instalações nativas e Docker é manual: o usuário copia `~/.local/share/vbm/vbm.db` para o volume. Sem migração de schema. Documentar isso no `CLAUDE.md` ou em uma nota no Dockerfile.

### Integrações externas
- Provider OpenAI-compat (`VBM_EMBED_URL`): se o daemon estiver em container e o LLM em `http://localhost:11434` (Ollama no host), o usuário precisa usar `host.docker.internal` ou network host. Documentar.
- Extensão Chrome não muda.

### Conflitos com decisões existentes
- **CLAUDE.md / "Nunca fazer bind em 0.0.0.0"**: a regra original é sobre o **default** do daemon, não sobre Docker. O escape hatch `VBM_BIND` está documentado no próprio comentário de `server.go` ("Docker only — e.g. VBM_BIND=0.0.0.0"). Esta CR formaliza o uso. Atualizar a frase no CLAUDE.md para "Default 127.0.0.1; só usar 0.0.0.0 dentro de container quando o port mapping do host estiver restrito ao loopback."
- **D-002 / D-003 / D-004 / D-005**: nenhum impacto.

### CRs relacionadas
- CR-0001 — `internal/llm` precisa do `VBM_EMBED_URL`; em Docker, atenção pra `host.docker.internal` se Ollama está no host.
- CR-0004 — `envSpecs` precisa receber `VBM_DATA_DIR`.

## Decisões necessárias antes de implementar

1. **Caminho do volume** — proposto: `/data`. Bem estabelecido como convenção.
2. **Versão da imagem distroless** — `gcr.io/distroless/static-debian12:nonroot`. Tag `:nonroot` força UID não-root (65532). Sem novidade.
3. **Tag default da imagem** — proposto: `vbmd:dev` em build local. Sem registry oficial nesta CR.
4. **Healthcheck no Dockerfile** — proposto: `HEALTHCHECK CMD ["/vbmd", "healthz"]`. Mas `vbmd` hoje só tem subcomando `server`. Adicionar `vbmd healthz` que faz HTTP GET local seria escopo extra. **Proposta**: omitir HEALTHCHECK nesta CR; quem quiser pode adicionar via `docker run --health-cmd`.

Decisões 1–4 podem ser tomadas durante a implementação se não houver opinião forte.

## Critério de aceite

1. `cd daemon && docker build -t vbmd:dev .` completa em <2min sem erros.
2. `docker images vbmd:dev` mostra tamanho final ≤15MB.
3. `docker run --rm -p 127.0.0.1:7532:7532 -e VBM_BIND=0.0.0.0 -e VBM_DATA_DIR=/data -v $(pwd)/.docker-data:/data vbmd:dev` sobe e responde `curl 127.0.0.1:7532/healthz` → `{"ok":true}`.
4. Banner do startup (CR-0004) mostra `VBM_DATA_DIR [SET]`.
5. Indexar uma página via extensão (apontando a extensão pra `127.0.0.1:7532`) cria entrada em `./.docker-data/vbm.db` no host.
6. Reiniciar o container preserva os dados (mesmo volume).
7. Imagem roda como UID não-root (verificável com `docker run --rm vbmd:dev id` — *não funciona* porque distroless não tem `id`; alternativa: `docker inspect vbmd:dev | grep User` mostra `nonroot:nonroot`).
8. `go test ./...` continua verde (mudança em `nm.DataDir` é trivial).
9. CORS continua aceitando `chrome-extension://*` — sem mudança no router.

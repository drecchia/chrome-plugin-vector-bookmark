# CR-0002 — Modos adicionais de indexação: seleção, transcrição e comentários do YouTube

**Data:** 2026-05-02
**Solicitante:** Danilo Recchia
**App Version:** extension 0.1.0
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Extension popup (UX), Extension content script (extração), Daemon HTTP (validação de novos modes)

---

## Descrição da mudança

Adicionar **três novos modos de indexação** ao painel do popup, complementando os 4 já existentes:

1. **`selection`** — indexa apenas o texto selecionado pelo usuário na página (`window.getSelection()`).
2. **`yt_transcript`** — indexa a transcrição do vídeo do YouTube atual, raspada do próprio painel de legendas/CC do YouTube no DOM. Visível **apenas em URLs `youtube.com/watch`**.
3. **`yt_comments`** — indexa os top-N comentários **já visíveis no DOM** (sem auto-scroll). Visível apenas em `youtube.com/watch`.

Os 3 novos modos aparecem como radios adicionais no painel inline já implementado em CR-0001.

## Motivação

- **Seleção**: o usuário muitas vezes só quer guardar um trecho específico (citação, parágrafo) em vez do conteúdo inteiro. Hoje teria que copiar e colar no modo `manual`.
- **YouTube transcript**: vídeos são primeira-classe pra "memória de navegação" mas hoje só indexam título + descrição. A transcrição contém o conteúdo real e é facilmente recuperável via FTS.
- **YouTube comments**: para vídeos técnicos/tutoriais, o conteúdo dos comentários frequentemente complementa ou corrige o vídeo. Indexar os mais visíveis dá uma boa amostra sem custo.

## Comportamento atual

- Painel do popup tem 4 modos: `full_text`, `llm_summary`, `manual`, `meta_only`.
- Conteúdo selecionado pelo usuário é ignorado.
- YouTube é tratado como qualquer outro site: Readability extrai a página inteira (geralmente boilerplate da UI do YT, não o conteúdo).

## Comportamento desejado

### Popup (em qualquer site)
- Novo radio **"Index selected text"** disponível sempre que `window.getSelection()` poderia retornar algo (na prática: sempre visível; o erro acontece no submit se vazio).

### Popup (apenas em `*.youtube.com/watch*`)
- Dois radios extras visíveis condicionalmente:
  - **"YouTube — transcribe (CC)"**
  - **"YouTube — comments (visible)"**
- Painel detecta hostname/path no `useEffect` que já lê a aba ativa.

### Content script
- Novo handler para `force_extract` com `intent: 'selection' | 'yt_transcript' | 'yt_comments'`.
- `selection`: retorna `window.getSelection()?.toString()` em uma `page_viewed` com `text` preenchido. Se vazio, o handler responde `{ ok: false, error: 'Nothing selected' }`.
- `yt_transcript`: tenta abrir o painel "Show transcript" do YouTube e raspar `<ytd-transcript-segment-renderer>` (ou o seletor estável corrente). Se não disponível, retorna erro "Transcript not available for this video".
- `yt_comments`: percorre os `<ytd-comment-thread-renderer>` já presentes no DOM, extrai `#content-text` de cada, junta com newlines.

### Service worker
- Mensagem `popup_force_index` ganha campo `intent?: 'selection' | 'yt_transcript' | 'yt_comments'`.
- Pipeline reutiliza o mesmo `pendingIngest` map: SW envia `force_extract` com `intent`, content script faz a extração específica e devolve `page_viewed` com `text`. Daemon recebe `mode: "manual"` (set-mode tags + texto que veio pronto), pois do ponto de vista do daemon não importa de onde veio o texto.

### Daemon
- **Sem mudança estrutural**. O daemon continua aceitando os 4 modes atuais. Os novos "modos" são uma camada de extração no client; o daemon vê apenas `mode: "manual"` para os 3 novos casos. Isso mantém o backend simples e evita schema/api churn.

## Análise de impacto

### Entidades afetadas
Nenhuma mudança em entidades persistidas. Esquema SQLite intacto.

### Backend
Sem mudanças. Os novos modos não chegam ao daemon — ele recebe ingest com `mode: "manual"` e o texto já extraído.

### Frontend / Extensão

- `extension/src/content/extract.ts`:
  - Novo handler para o tipo `force_extract` com payload `{ intent }`.
  - Função `extractSelection()`: `window.getSelection()?.toString().trim()`.
  - Função `extractYouTubeTranscript()`: clica/abre o painel de transcript via DOM, lê segmentos. Retry curto se o painel ainda está animando.
  - Função `extractYouTubeComments(maxN)`: itera nodes já carregados, sem scroll. Default N=50.
  - Cada função monta um `page_viewed` com `text` apropriado.
- `extension/src/background/service-worker.ts`:
  - `pendingIngest` ganha campo `intent?` opcional. Não muda a lógica de ingest no SW; apenas é repassado ao content script via `force_extract`.
  - Para `intent === 'selection'`: SW pode (alternativa) já saber o texto se quiser ler `chrome.scripting.executeScript` com `() => window.getSelection()?.toString()`. Mais robusto que round-trip via content script. Optar por essa via.
- `extension/src/popup/App.tsx`:
  - Estender o array de modos no painel para incluir os 3 novos labels.
  - Detectar YouTube watch via `currentTabUrl` e renderizar os dois radios YT condicionalmente.
  - Renomear o tipo `IngestMode` no popup para um union mais largo (apenas no nível do client), ou manter `mode: 'manual'` no envio e usar um campo separado `intent` para distinguir a origem. **Decisão recomendada**: campo `intent` separado, mantém `IngestMode` em `proto/types.ts` inalterado.
- `extension/proto/types.ts`:
  - Pode adicionar tipo auxiliar `ExtractIntent` apenas para mensagens internas SW↔content script. Não vai pro daemon.
- `extension/manifest.json`:
  - Verificar se permissões existentes (`activeTab`, `scripting`) bastam — provavelmente sim. Se não, adicionar nada (já temos).

### Dados existentes
Não requer migração.

### Integrações externas
Nenhuma. Toda a extração acontece localmente no DOM da aba ativa.

### Conflitos com decisões existentes

- **D-003 — Modos de ingest expostos como enum no client**: o backend continua expondo apenas 4 modes. Os 3 novos são tipos de extração no client que se traduzem em `mode: "manual"` no envio. Isso preserva a decisão sem ampliá-la — o critério "ruído de boilerplate, custo, foco" continua aplicado.
- **CR-0001**: o painel inline de "Index this site now" continua sendo o único entry point. Set-mode de tags continua aplicando.

## Decisões necessárias antes de implementar

1. **Quantos comentários no `yt_comments`** — proposto N=50 (suficiente pra capturar os primeiros threads sem ficar pesado). Ajustar se preferir 20 ou 100.
2. **Seletor da transcrição do YouTube** — YouTube muda seletores periodicamente. Proposta: começar com `ytd-transcript-segment-renderer .segment-text`, com fallback genérico para `[role="button"][aria-label*="ranscript"]` para abrir o painel. Aceitar que possa quebrar com refactor de UI do YT — adicionar log claro quando seletor falha.
3. **Posição dos radios YouTube no painel** — proposto: aparecem ao final dos 4 atuais, com cabeçalho `"— YouTube"` separando-os visualmente.

Decisões 1–3 podem ser tomadas durante a implementação se não houver opinião forte.

## Critério de aceite

1. Painel do popup mostra um radio extra **"Index selected text"** em qualquer site.
2. Selecionar um parágrafo numa página, abrir popup, escolher "Index selected text", clicar Confirm → o conteúdo do chunk indexado contém **apenas** o texto selecionado (verificável em `SELECT text FROM chunks WHERE page_id=…`).
3. Em uma página `youtube.com/watch?v=…`, dois radios adicionais aparecem: "YouTube — transcribe (CC)" e "YouTube — comments (visible)".
4. Em uma página fora do YouTube, esses dois radios **não** aparecem.
5. Em vídeo do YouTube com legendas habilitadas: confirmar com "transcribe" → chunk indexado contém os segmentos da transcrição.
6. Em vídeo sem legendas: flash vermelho "Transcript not available for this video"; nada é indexado.
7. Em vídeo com comentários carregados: confirmar com "comments" → chunk indexado contém os top-N comentários visíveis (verificar contagem ≥1 e ≤N).
8. Selecionar texto vazio e tentar indexar → flash vermelho "Nothing selected".
9. `go test ./...`, `npm run typecheck`, `npm run build` continuam verdes.
10. Daemon **não** recebe novos `mode` values — `curl /pages` para uma página YT mostra a entrada existente com `mode` interno mapeado para `manual` (ou pelo menos sem erro no log).

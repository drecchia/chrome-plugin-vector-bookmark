# CR-0003 — Botão "Sugerir tags" via LLM no popup

**Data:** 2026-05-02
**Solicitante:** Danilo Recchia
**App Version:** extension 0.1.0
**Status:** implemented
**Urgência:** low
**Domínios afetados:** Extension popup (UX), Extension content script (Readability), Daemon HTTP (novo endpoint), Daemon LLM client

---

## Descrição da mudança

Adicionar um **botão "✨" (ou "Suggest")** ao lado do input de tags CSV no painel "Index this site now" do popup. Ao clicar:

1. Content script extrai o texto da página (Readability) + meta tags + título.
2. Popup envia esse contexto ao daemon, junto com a lista atual de tags conhecidas (das que o usuário já tem no banco).
3. Daemon chama o LLM via `internal/llm` com prompt instruindo "gere até 3 tags curtas, reusando a lista existente quando aplicável".
4. Resposta volta ao popup, que **acrescenta** as tags geradas ao input CSV existente (dedup por comparação normalizada — sem repetir o que o usuário já digitou ou o que veio do banco).

## Motivação

- Hoje o usuário precisa digitar tags manualmente. Isso desincentiva uso e gera taxonomia inconsistente (`read-later` vs `readlater` vs `read_later`).
- Já temos `internal/llm` instalado (CR-0001). Reusá-lo para sugestão de tags é barato — uma chamada `chat/completions` curta.
- Mantém **reuso de taxonomia**: o LLM recebe a lista de tags já usadas no banco e o prompt pede "prefer existing tags when relevant".

## Comportamento atual

- Painel do popup tem campo CSV de tags livremente digitado, com `<datalist>` sugerindo tags já usadas (autocomplete passivo). Sem geração automática.

## Comportamento desejado

### Popup
- Input de tags vira uma `<div>` flex com 2 elementos:
  - O `<input>` CSV (largura ajustada).
  - Botão `[✨]` à direita (24-28px), com tooltip "Suggest tags via LLM".
- Click no botão:
  1. Mostra estado "loading" (botão fica desabilitado, ícone vira spinner ou "…").
  2. Triggera extração no content script (`force_extract` com `intent: 'suggest_tags'` — content script extrai mas **não envia `page_viewed`**, retorna direto via callback).
  3. Recebe `{title, text, meta}` no SW.
  4. SW envia `POST /tags/suggest` ao daemon com `{text, title, meta}`.
  5. Daemon roda LLM, retorna `{tags: ["foo", "bar", "baz"]}`.
  6. Popup acrescenta tags ao `tagsCSV` state com dedup normalizado, e flash verde "Suggested 3 tags".
- Erros (LLM offline, conteúdo insuficiente, sem CC etc.) → flash vermelho com mensagem específica.

### Daemon
- Novo endpoint `POST /tags/suggest`:
  ```json
  { "title": "...", "text": "...", "url": "...", "meta": {...} }
  ```
  - Carrega lista de tags existentes via `Store.ListTags()`.
  - Chama `llm.SuggestTags(ctx, title, text, meta, existingTags)` (novo método).
  - Retorna `{ "tags": ["foo", "bar"] }` (1-3 entries, normalizadas via `normalizeTag`).
- Sem LLM configurado → 503 com `{"error":"LLM not configured"}`.
- Conteúdo insuficiente (texto < 100 chars) → 400 com `{"error":"not enough content"}`.

### Content script
- Novo `intent: 'suggest_tags'` no listener `force_extract`. Não emite `page_viewed`; em vez disso, **devolve o payload via `sendResponse`**:
  ```ts
  { ok: true, payload: { title, text, meta } }
  ```
- O SW intercepta esse payload e o usa para chamar o endpoint, sem tocar no badge nem no fluxo de ingest.

### LLM
- Novo método `Client.SuggestTags(ctx, title, text, meta, existing)` em `internal/llm/llm.go`.
- Prompt fixo: "You generate tags for retrieval/curation. Output between 1 and 3 short tags (1-2 words each, lowercase, kebab-case). Reuse from this existing list when applicable: [...]. Otherwise create new ones aligned with the same style. Output JSON only, format `{\"tags\":[\"tag1\",\"tag2\"]}`."
- `temperature: 0.3`. Truncar `text` no mesmo `maxInputChars=32_000`.
- Parser tolerante: aceita JSON puro ou JSON dentro de fences markdown. Filtra tags inválidas via `normalizeTag`. Trunca em 3.

## Análise de impacto

### Entidades afetadas
Nenhuma. Sem schema novo.

### Backend
- `daemon/internal/llm/llm.go` — novo método `SuggestTags(ctx, title, text, meta, existing) ([]string, error)`.
- `daemon/internal/server/routes.go` — novo handler `POST /tags/suggest`.
- Sem mudança em store, queue, schema.

### Frontend / Extensão
- `extension/proto/types.ts` — novo tipo de request/response opcional para `/tags/suggest` (apenas para clareza de tipos no client).
- `extension/src/background/daemon-client.ts` — nova função `suggestTags(payload)`.
- `extension/src/background/service-worker.ts` — novo handler `popup_suggest_tags`: dispara `force_extract` com `intent: 'suggest_tags'`, recebe payload, chama `suggestTags`, devolve ao popup.
- `extension/src/content/extract.ts` — novo branch `intent === 'suggest_tags'`: extrai via Readability + meta + título, retorna via `sendResponse` (não emite `page_viewed`).
- `extension/src/popup/App.tsx` — botão ✨ ao lado do input. Estado `suggestLoading`. Handler `handleSuggestTags`. Função utilitária `mergeTagsCSV(existing, generated)`.

Não requer migração.

## Decisões necessárias antes de implementar

1. **Limite máximo de tags** — proposto: 3 (alinhado ao pedido do usuário). Hardcoded no prompt + truncar no client.
2. **Tamanho do contexto enviado ao LLM** — proposto: truncar `text` em 8000 chars (suficiente pra um bom resumo temático, mantém a chamada barata). O `maxInputChars=32_000` do `Summarize` é exagero pra esse caso.
3. **Comportamento quando o LLM retorna 0 tags válidas** — proposto: flash "Could not suggest tags — try again" sem alterar o input.

Decisões 1–3 podem ser tomadas durante a implementação se não houver opinião forte.

---

## Critério de aceite

1. Painel do popup mostra um botão ✨ ao lado do input de tags.
2. Click no botão em uma página com texto suficiente (>100 chars Readability) chama o LLM e popula o campo com 1–3 tags em ≤5s.
3. Se o campo já contém `read-later, ai`, e o LLM retorna `["ai", "machine-learning", "python"]`, o resultado final no campo é `read-later, ai, machine-learning, python` (sem duplicar `ai`).
4. Click em página sem texto suficiente → flash vermelho "Not enough content to suggest tags".
5. LLM offline (`VBM_EMBED_URL` ausente ou `/chat/completions` falha) → flash vermelho "LLM not configured" ou "LLM call failed".
6. Tags geradas passam pelo `normalizeTag` no daemon (lowercase, kebab-case-friendly, max 64 chars).
7. Lista de tags existentes (`/tags`) é usada como contexto: se o banco tem `read-later`, o LLM tende a reusar exatamente esse termo em vez de criar `read_later` ou `readlater`.
8. `go test ./...`, `npm run typecheck`, `npm run build` continuam verdes.
9. Endpoint `POST /tags/suggest` é callable via curl com payload de exemplo e retorna JSON `{tags: [...]}`.

# CR-0008 — Bearer-token auth opcional cobrindo HTTP e WebSocket

**Data:** 2026-05-09
**Solicitante:** Danilo Recchia
**App Version:** daemon 0.1.0 / extension 1.0.0
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Daemon HTTP/WebSocket, Extensão (config + cliente HTTP), docs

---

## Descrição da mudança

Adicionar autenticação opcional via Bearer token no daemon, cobrindo HTTP e WebSocket. Configurada por env var `VBM_AUTH_TOKEN` no backend e por um campo "Auth token" na seção Settings do popup. Vazio = comportamento atual (acesso aberto).

## Motivação

O default do daemon é bind em `127.0.0.1`, então em uso single-user não há risco. Mas em deploys que expõem a porta publicamente — Docker com port mapping não-loopback, K8s `Service`, reverse proxy — qualquer cliente que alcance a porta consegue ler/escrever no banco. Sem auth, isso bloqueia esses cenários. O ajuste é mínimo: opt-in, sem schema novo, sem rotação, sem hash.

## Comportamento

- `VBM_AUTH_TOKEN` vazio/ausente → middleware é no-op (preserva setups locais).
- `VBM_AUTH_TOKEN=<valor>` → todas as rotas exigem o token, exceto `/healthz` e `/metrics` (probes/scrape).
  - HTTP: header `Authorization: Bearer <token>`.
  - WebSocket (`/ws`): browsers não enviam header custom no handshake, então `?token=<token>` na query é aceito como fallback.
- Extensão: novo campo "Auth token (optional)" no popup, persistido em `chrome.storage.local` como `vbmAuthToken`, injetado em todas as fetches via `Authorization: Bearer …`.

## Arquivos modificados

### Daemon
- `daemon/internal/server/server.go`:
  - `envSpecs`: adicionar linha `{"VBM_AUTH_TOKEN", "no auth (open access)"}`.
  - `Run()`: ler `os.Getenv("VBM_AUTH_TOKEN")` e propagar para `newRouter`.
- `daemon/internal/server/routes.go`:
  - Assinatura: `newRouter(..., authToken string)`.
  - `authMiddleware`: early-return quando `token == ""`; aceita `Authorization: Bearer <token>` ou `?token=<token>`.
  - Conectado dentro do `r.Group(...)` (após `corsMiddleware`). `/healthz` e `/metrics` ficam fora do Group → permanecem públicos.

### Extensão
- `extension/src/background/native-bridge.ts`: campo `authToken` em `DaemonConfig`, persistido em `vbmAuthToken`. Helper `authHeader(cfg)`.
- `extension/src/background/daemon-client.ts`: todas as 13 fetches passam `Authorization: Bearer …` quando token presente.
- `extension/src/popup/App.tsx`: input password (com toggle Show/Hide) na seção Settings, abaixo do par host/port.

### Docs

## Decisões tomadas

1. **Token vazio = sem auth (compat)** ao invés de bloquear o startup — preserva setups 127.0.0.1-only existentes. Decisão registrada em D-NNN se relevante; aqui só por economia de mudança.
2. **Sem rotação, sem hash, sem rate-limit** — fora do escopo "ajuste mínimo".
3. **`/healthz` e `/metrics` continuam públicos** — necessário para liveness probes e Prometheus scrape; mantido como antes.
4. **Query `?token=` apenas como fallback de WS** — necessário porque browsers não suportam headers custom no handshake WS. Risco aceito: query strings podem leak em logs de proxy intermediário (documentado).

---

## Critério de aceite

1. Sem `VBM_AUTH_TOKEN`: `curl http://127.0.0.1:7532/status` → 200; extensão funciona normalmente.
2. Com `VBM_AUTH_TOKEN=abc123`:
   - `curl http://127.0.0.1:7532/status` → 401 com `{"error":"unauthorized"}`.
   - `curl -H "Authorization: Bearer abc123" .../status` → 200.
   - `curl http://127.0.0.1:7532/healthz` → 200 (público).
3. WS: `wscat -c ws://127.0.0.1:7532/ws` → 401. `wscat -c "ws://127.0.0.1:7532/ws?token=abc123"` → conecta e recebe `{"type":"status",...}`.
4. Extensão com token salvo via popup → todas as features (search, ingest, suggest tags, blacklist) funcionam contra daemon protegido.
5. Banner de startup do daemon lista `VBM_AUTH_TOKEN` como `[SET]` ou `[MISSING]`.
6. `go test ./internal/server/...` verde — `routes_test.go` (que usa token não-vazio) continua passando.

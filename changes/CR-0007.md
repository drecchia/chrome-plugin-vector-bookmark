# CR-0007 — Timeline: gráfico cobre todos os dias do período + lista filtrada por dia

**Data:** 2026-05-08
**Solicitante:** Danilo Recchia
**App Version:** daemon 0.1.0 / extension 0.1.0
**Status:** implemented
**Urgência:** medium
**Domínios afetados:** Daemon HTTP UI (aba Timeline), Daemon `/history` endpoint, Daemon SQLite store

---

## Descrição da mudança

Duas correções relacionadas na aba **Timeline** da UI servida pelo daemon (`http://127.0.0.1:7532/`):

1. **Gráfico passa a cobrir todos os dias do período (semana ou mês)**, com altura proporcional ao volume real de páginas em cada dia. Hoje só os ~3 dias mais recentes aparecem com barra colorida — o resto fica zerado mesmo havendo tráfego.

2. **Lista abaixo do gráfico passa a ser filtrada por um único dia**: por padrão, o dia atual; ao clicar numa barra do gráfico, troca para o dia clicado. A barra do dia selecionado fica destacada visualmente.

## Motivação

**Parte 1 — gráfico truncado**: a causa é que o handler `/history` constrói o map `daily` a partir das linhas paginadas (`LIMIT 100`). Em períodos com tráfego suficiente, o limite esgota antes de chegar aos dias mais antigos da semana, então `daily[k]` fica em 0 para esses dias e o gráfico mostra apenas barras pequenas/cinza nos dias mais antigos. O usuário interpreta como "só carrega os últimos 3 dias", o que é coerente com o sintoma.

**Parte 2 — lista verbosa**: a lista atual mostra todas as páginas da semana (até 100) agrupadas por data. Para semanas movimentadas, isso vira um wall-of-text. Com o gráfico já oferecendo a visão agregada por dia, faz mais sentido a lista mostrar **só o dia selecionado** — coerente com o gesto natural de "quero ver o que fiz nessa quarta-feira".

## Comportamento atual

### Gráfico (`htBuildChart` em `routes.go:726`)
- Renderiza N barras (7 para semana, até 31 para mês), iterando sobre todos os dias do período.
- Altura de cada barra = `daily[k]` / `maxVal`. Cor: `#6366f1` se `v>0`, `#e5e7eb` se `v==0`.
- `daily[k]` vem do daemon. Quando o daemon perde dias (por causa do limit), barras desses dias ficam como `v=0` (cinza, altura zero) ou `v=2` (mínimo de 2px sobre `maxVal`).

### Backend `/history` (`routes.go:1538`)
- `histRows := s.GetPageHistoryRows(fromMs, toMs, limit)` — `LIMIT 100` em `pages ORDER BY visit_ts DESC` (`sqlite.go:849`).
- Constrói `daily` no loop sobre `pageOrder` (`routes.go:1611`): `daily[dateKey]++` para cada page retornada.
- Resultado: `daily` reflete só as 100 páginas mais recentes, não o total real do período.

### Lista (`htRender` em `routes.go:748`)
- Agrupa `pages` por `toLocaleDateString(...{weekday,month,day})`.
- Renderiza um `<div class="hist-group">` por dia, com header da data e lista de páginas.
- Sem interação com o gráfico — clicar numa barra é no-op (só tem `<title>` SVG nativo).

## Comportamento desejado

### Gráfico
- Continua renderizando todas as N barras do período.
- **Heights vêm de uma agregação SQL real** (count de pages por dia em todo o período), independente de `limit`.
- Cada barra ganha `data-day="YYYY-MM-DD"` e `cursor:pointer`.
- A barra correspondente ao **dia selecionado** fica destacada (cor mais escura, ex.: `#4338ca`, ou stroke); barras com `v>0` continuam `#6366f1`; barras com `v==0` continuam cinza `#e5e7eb` mas ainda clicáveis (lista vai mostrar "Sem páginas neste dia").

### Lista
- Agora exibe **apenas as páginas do dia selecionado**, em ordem cronológica.
- Header pequeno acima da lista mostra o dia selecionado em forma legível (ex.: "Quarta, 8 de maio").
- Sem agrupamento por data (porque é sempre um único dia).

### Estado e fluxo
- Estado JS novo: `htSelectedDay: string` no formato `YYYY-MM-DD` (UTC).
- Default ao entrar na aba Timeline: hoje (UTC), clamped:
  - Se hoje cai dentro do período → seleciona hoje.
  - Senão → seleciona o último dia do período com `daily[k] > 0`; se não houver, seleciona o último dia do período.
- Navegação Prev/Next/Week/Month: aplica a mesma regra de default ao novo período.
- Click numa barra: atualiza `htSelectedDay`, re-renderiza só o destaque do gráfico + a lista (sem refetch dos counts diários).

### Endpoint `/history`
- Continua aceitando os mesmos params (`from`, `to`, `limit`).
- `daily` no response passa a ser **a contagem real de páginas por dia em [from, to)**, computada via SQL agregada — não derivada do array de pages limitado.
- `pages` continua paginado por `limit`. Para a lista do dia, o frontend chama `/history?from=<dayStart>&to=<dayEnd>&limit=500`.

## Análise de impacto

### Entidades afetadas
Nenhuma. Sem mudança de schema. Apenas leitura agregada nova sobre `pages.visit_ts`.

### Backend (Go)
- `daemon/internal/store/sqlite.go`:
  - Função nova `GetDailyPageCounts(fromMs, toMs int64) (map[string]int, error)`. Query: `SELECT strftime('%Y-%m-%d', visit_ts/1000, 'unixepoch') AS day, COUNT(*) FROM pages WHERE visit_ts >= ? AND visit_ts < ? GROUP BY day`. Retorna mapa `YYYY-MM-DD -> count`.
- `daemon/internal/server/routes.go` (handler `/history` em `routes.go:1538`):
  - Substituir o `daily := make(map[string]int)` montado no loop sobre `pageOrder` (linhas 1595, 1611-1612) por chamada a `s.GetDailyPageCounts(fromMs, toMs)`.
  - Remover `daily[dateKey]++` no loop de pages.
  - Em caso de erro do `GetDailyPageCounts`, retornar 500 com `{"error":"history query failed"}` (mesmo padrão do `GetPageHistoryRows`).

### Frontend (JS embedded em `routes.go`)
- `daemon/internal/server/routes.go` bloco `// ── history timeline ──` (linhas 711-809):
  - Adicionar var de estado `htSelectedDay`.
  - Helper `htClampSelectedDay(daily, fromMs, toMs)` aplica a regra de default (hoje > último com tráfego > último do período).
  - `htRender()` passa a fazer **dois fetches**:
    1. `/history?from=<periodStart>&to=<periodEnd>&limit=1` — usa só `d.daily` para o gráfico; `d.pages` é descartado (limit=1 minimiza payload).
    2. Após determinar `htSelectedDay`, fetch `/history?from=<dayStart>&to=<dayEnd>&limit=500` para popular a lista.
  - `htBuildChart` recebe `htSelectedDay`; cada `<rect>` ganha `data-day="..."` e cor conforme regra acima.
  - Listener delegado em `#ht-chart`: clique em `rect[data-day]` → `htSelectDay(day)`.
  - `htSelectDay(day)`: atualiza `htSelectedDay`, re-render do chart (sem refetch) + fetch da lista do dia.
  - Listas de Prev/Next/Week/Month: ao trocar período, resetar `htSelectedDay` via `htClampSelectedDay` após o primeiro fetch terminar.
  - Renderização da lista: removida o agrupamento por data (`groups`/`order`), substituído por uma lista flat das páginas do dia, com header `<div class="hist-selected-day">` mostrando o dia selecionado em forma legível.

### Dados existentes
Não requer migração. Páginas existentes continuam como estão; a mudança é só de leitura/visualização.

### Integrações externas
Nenhuma.

### Conflitos com decisões existentes
- Nenhum conflito identificado. A mudança não altera contratos públicos do daemon (`/history` continua compatível — o shape do response é o mesmo, só `daily` fica mais preciso).

### CRs relacionadas
- Nenhuma direta. A aba Timeline atual não tem CR específica de origem.

## Decisões necessárias antes de implementar

1. **Timezone do agrupamento por dia** — proposto: **UTC** (consistente com o que o daemon já faz em `routes.go:1611` via `time.UnixMilli(...).UTC().Format("2006-01-02")`). Mudar para timezone local do servidor seria possível mas inconsistente com o resto. UTC é suficiente para POC.
2. **Cor da barra selecionada** — proposto: `#4338ca` (versão mais escura do `#6366f1` atual). Sem stroke separado.
3. **Comportamento ao clicar numa barra com `v==0`** — proposto: **permitir click**, mostrar lista vazia com mensagem "Nenhuma página neste dia". Coerente com o resto da UI.
4. **Limit do fetch do dia** — proposto: 500 (mesmo limite do daemon). Para POC um único dia raramente passa disso.

Decisões 1–4 podem ser tomadas durante a implementação se não houver opinião forte.

## Critério de aceite

1. Após indexar 50+ páginas distribuídas em ≥5 dias da semana atual, abrir `http://127.0.0.1:7532/` → tab **Timeline**:
   - **Todas as 7 barras** da semana renderizadas com altura proporcional ao tráfego de cada dia (não só as 3 mais recentes).
   - Barra de **hoje** destacada em cor mais escura.
   - Lista abaixo mostra **só páginas de hoje**, sem agrupamento por dia, com header pequeno "Quarta, 8 de maio" (ou equivalente no idioma do browser).
2. Clicar numa barra de outro dia → lista atualiza para aquele dia, highlight do gráfico muda. Não refaz o fetch dos counts diários.
3. Clicar numa barra de um dia sem páginas → lista mostra "Nenhuma página neste dia"; highlight muda corretamente.
4. Botões **Prev/Next** voltam/avançam semana e o highlight reseta:
   - Se a janela inclui hoje → seleciona hoje.
   - Senão → seleciona o último dia da janela com tráfego, ou o último dia se nenhum tiver tráfego.
5. **Month** mode: até 31 barras com mesma lógica; click em qualquer barra filtra a lista.
6. `curl 'http://127.0.0.1:7532/history?from=<weekStart_ms>&to=<weekEnd_ms>&limit=1' | jq .daily` → confirmar que o map cobre todos os dias com tráfego, mesmo com `limit=1` (regra antes da CR: cobertura quase zero).
7. `go test ./...` continua verde.
